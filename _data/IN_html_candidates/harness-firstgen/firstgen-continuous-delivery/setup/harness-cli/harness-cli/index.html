Harness CLI Features

You can use the Harness CLI to do the following:

  • Authenticate:
    • Log into your Harness account using username and password. Currently, Harness CLI does not support the other authentication methods supported by Harness Manager, such as LDAP, SSO, and SAML.
  • Deploy:
    • Deploy a Pipeline or Workflow.
    • List deployment info such as Application, Workflow, Pipeline, and Workflow variables.
  • List:
    • List your Applications, Services, Environments, Infrastructure Definitions, Workflows, and Pipelines.
  • Approve:
    • List, approve, reject, and add comments to pending Approvals.
  • Automate:
    • Write scripts to automate Harness operations covered by the Harness CLI commands.

What Can't I do with Harness CLI?

Currently, the Harness CLI has the following limitations:

  • Manage or list Account and Shared Resources information, such as Users, SSO settings, Cloud Providers, Connectors, Templates, Alerts, Delegates, and Tags.
  • Create or modify Application components.
  • Monitor Continuous Verification.

Install the Harness CLI using Homebrew

To install the Harness CLI on a Mac using Homebrew, do the following:

  1. Install or update (brew upgrade) Brew.
  2. Tap the Harness repository: brew tap wings-software/homebrew-tap
  3. Install the Harness CLI: brew install harness
    1. If desired, upgrade the Harness CLI: brew upgrade harness

You will see output similar to: harness has been installed as /usr/local/bin/harness

Enter harness --version to view the version number of the Harness CLI.

The harness --help command displays the CLI help:

Command Line Utility for Harness Operations

Usage:
  harness [command]

Available Commands:
  approve     Approve a pending approval corresponding to a workflow/pipeline
  deploy      Deploys a workflow/pipeline corresponding to an application
  get         Retrieves data about a specified entity
  help        Help about any command
  login       Enables the user to login to the harness CLI
  reject      Reject a pending approval corresponding to a workflow/pipeline

Flags:
  -h, --help      help for harness
      --version   version for harness

Requirements

Currently, Harness distributes its CLI using the Homebrew package manager on Mac.

The binaries for Harness CLI are available on Github for all operating systems: https://github.com/wings-software/harness-cli-binaries.

Download the Harness CLI Binary using cURL

To download the Harness CLI binary on Mac/Linux, use the following cURL command:

curl https://raw.githubusercontent.com/wings-software/harness-cli-binaries/master/harnesscli_1.0.0_Linux_x86_64.tar.gz > harnesscli.tar.gz

Ensure you set the PATH variable, or copy the binary to the bin directory.

You can extract the TAR file using tar -zxvf harnesscli.tar.gz.

To start using Harness CLI, simply run ./harness.

CLI Authentication

Currently, Harness CLI does not support all of the authentication methods supported by Harness Manager, such as LDAP, SSO, and SAML. You can only authenticate using a username (email address) and password.

Use your Harness username (registered email address) and password with the harness login command to log into your Harness account:

harness login -u "johndoe@harness.io" -p "mypassword"
Logged in Successfully to app.harness.io
You can also enter harness login and you will be prompted for the username and password.

Once you are logged in, you can use the Harness CLI commands described in this document.

For Harness On-Prem, you might also want to use the -d option to specify a domain. For example, if your On-Prem domain is named internal.harness.io, you would log in like this:

harness login -u "johndoe@harness.io" -p "mypassword" -d "internal.harness.io"

Command Reference

This section is a reference for the Harness CLI commands. You can use harness --help to view commands in the CLI help.

harness login

Logs into your Harness account.

Usage

harness login <options> [parameters]

Options

Option and Shorthand Syntax

Description

--username , -u

Username

--password , -p

Password

--domain, -d

Change the login domain. This is used for Harness On-Prem. It is not needed for SaaS logins.

Examples

harness login

harness login -u "johndoe@harness.io" -p "mypassword"

harness login -u "johndoe@harness.io" -p "mypassword" -d "onprem.harness.io"

Sample Output

Logged in Successfully

harness get

List information on Harness Application components and deployments.

Usage

harness get [subcommand]

Subcommands and Options

Subcommand

Description

applications

Lists all Applications.

Example: harness get applications

approvals

Lists all pending approvals.

Example: harness get approvals

artifacts

Lists all artifacts corresponding to a specific Service.

Example: harness get artifacts -s <serviceId>

deployments

Lists the 20 most recent deployments.

Example: harness get deployments

services

Lists the Services for a specified Application The parameter -a specifies the Application for which Services are listed.

Example: harness get services -a <appId>

environments

Lists the Environments for a specified Application (specified using the parameter -a).

Example: harness get environments -a <appId>

infradefinitions

Only for Harness accounts migrated to Infrastructure Definitions.

Lists the Infrastructure Definitions corresponding to a specific Application and an Environment. Both Application ID and Environment ID are required.

Example: harness get infradefinitions -a <appId> -e <envId>

inframappings

Only for Harness accounts that use Service Infrastructures because they have not been migrated to Infrastructure Definitions. 

Lists the Service Infrastructures for a specific Application, Environment, and Service. Application ID, Environment ID, and Service ID are required.

Example: harness get inframappings -a <appId> -e <envId> -s <serviceId>

workflows

List the Workflows for a specified Application. The parameter -a specifies the Application for which Workflows are listed.

Example: harness get workflows -a <appId>

pipelines

Lists the Pipelines for a specified Application. The parameter -a specifies the Application for which Pipelines are listed.

Example: harness get pipelines -a <appId>

Examples

harness get applications

harness get workflows -a "xxxxSmUISimoRrJL6NL73w"

harness get services -a "xxxxSmUISimoRrJL6NL73w"

harness deploy

Deploys a Harness Workflow or Pipeline. This command is the same as the Start New Deployment dialog in a Workflow or Pipeline in the Harness Manager:

Just as with the Harness Manager, the harness deploy command requires that you specify an artifact to be deployed.

Usage

harness deploy <options> [parameters]

Options

You must specify a Harness Application, either a Workflow or Pipeline, and any artifacts.

Option and Shorthand Syntax

Description

--application, -a 

Application containing the Pipeline or Workflow you want to deploy.

--artifacts, -A  (string array)

Artifacts to be deployed.

--pipeline, -p

Pipeline to be deployed.

--workflow, -w

Workflow to be deployed.

--configFile, -c

Takes the config file path as the input.

Example: harness deploy --configFile /Users/johndoe/Desktop/deploy.yaml

For an example config file, see Examples below.

--variables, -v (string array)

Workflow variables in the format variable_name:variable_value

Examples

Deploy using typical commands:

harness deploy -a "VlsYedU4Q_2zD-GjLKAp-A" -p "wDRG-hGbQDus_onGYREXBw" --artifacts "hCA01UywRl62b57Vmh1Hxw" --artifacts "Rrz3BbixRWWJRpkTk3bl-g"

Deploy using config file:

harness deploy --configFile /Users/johndoe/Desktop/deploy.yaml

Here is a sample config file:

application: VlsYedU4Q_2zD-GjLKAp-A
workflow: Eh4MAqm1QX6wYH5XWeQpiA
artifacts:
  - hCA01UywRl62b57Vmh1Hxw
  - Rrz3BbixRWWJRpkTk3bl-g
variables:
  - environment:djsbr8kfbas9sebw
  - workflowvariable:variableValue

The config file has the following requirements:

  • The file must be YAML with the yaml extension.
  • The file may contain the same options and parameters you would enter in the command.
  • If required options and parameters are omitted, you will be prompted for missing options and parameters.
  • Only the options and parameters listed for harness deploy above may be used.
Sample Output
harness deploy -a "FjprBZJ4TPqDtxRePundnQ" -p "VvU2uB-kQOeDqf4SHkjd_w"
Execution ID: NRkmqlrNQ_-g-RIKITX2rA

You can paste the Execution Link into your browser to see the deployment in Harness Manager.

harness deploy info

Provide information about the Services and variables involved in the deployment. This command helps you identify which variables and artifacts to provide when running a deployment.

In the case of a templatized Workflow, where component information such as Service, Infrastructure Definition/Service Infrastructure, and Environment must be supplied at runtime, you must provide the template variables in harness deploy info.

Usage

harness deploy info <options> [parameters]

Options

Option and Shorthand Syntax

Description

--application, -a

Specifies the application containing the deployment.

--pipeline, -p

Specifies the Pipeline to be deployed.

--workflow, -w

Specifies the Workflow to be deployed.

--variables, -v

Specifies the Workflow variables in the format variable_name:variable_value.

Example

The following example provides information on a Workflow:

harness deploy info -a "FjprBZJ4TPqDtxRePundnQ" -w "TVOlkuhoR1mque7xiFqwmw"

The following example provides Service and Environment variable values for a templatized Workflow:

harness deploy info --application ZQx9DFlATai4KSAc7CHoyw --workflow f_KmcZoUS4Oz3gC3vs1nEw --variables "Environment:DCKdjNpWRUuHK9e9lP7Fww" --variables "Service:auzfiy6WR6qGmgvkC41hWA"

Sample Output

The following example provides information on a Workflow:

harness deploy info -a "FjprBZJ4TPqDtxRePundnQ" -w "TVOlkuhoR1mque7xiFqwmw"

Workflow does not have variables

Please provide an artifact for the following Services:

S.No.	Service Name	Service ID	

1	To-Do List K8s	QjRQNXtnRiO6Nb63CNSwSw

Automation using CLI

The Harness CLI is designed to work as a scripting tool where you can use Bash programming to construct an automated deployment script.

A variable in Bash can contain a number, a character, or a string of characters. You do not need to declare a variable. Simply assigning a value to its reference will declare it.

Piping Example

The following examples use piping to get the Workflows for an Application.

Example 1:

harness get workflows -a $(harness get applications | grep "Harness Sample App" | awk '{print $5}')
S.No.		Workflow Name			Workflow Type	Workflow ID				
1		To-Do List K8s Canary		CANARY		TVOlkuhoR1mque7xiFqwmw		
2		To-Do List K8s Rolling		ROLLING		A__CeH2jR72x9CK8a97PsQ

Example 2:

harness get workflows -a $(harness get applications | grep "Harness Sample App" | sed -E "s/.*Harness Sample App[[:blank:]]+//")
S.No.		Workflow Name			Workflow Type	Workflow ID

Advanced Example

The script example below does the following:

  1. Logs into Harness.
  2. Obtains an Application ID.
  3. Uses the Application ID to get the Workflow ID.
  4. Uses the Workflow ID to deploy the Workflow.
  5. Uses the resulting Execution ID to report the deployment's success.
#!/bin/sh
​
LOGIN=`harness login -u "admin@harness.io" -p "admin" -d "localhost:9090"`
APPID=` harness get applications | grep "Harness Sample App" | sed -E "s/.*Harness Sample App[[:blank:]]+//" `
echo "*************************************************"
echo "Fetching ApplicationId"
echo "ApplicationId: " $APPID
echo " "
 
echo "Fetching OrchestrationId"
ORCHESTRATIONID=` harness get workflows -a $APPID | grep -i "multi-artifact" | sed -E "s/.*multi-artifact[[:blank:]]+//" | awk '{print $2}'`
echo "OrchestrationId: " $ORCHESTRATIONID
echo " "
 
echo "Triggering Depoyment"
EXECUTIONID=`./harness deploy -a $APPID -w $ORCHESTRTIONID | grep "Execution ID:" | sed -E "s/.*Execution ID:[[:blank:]]+//" | awk '{print $1}'`
echo "Deployment Triggered"
echo "OrchestrationId: " $ORCHESTRTIONID
echo "Deployment completed Successfully"
echo " "
 
./harness get deployments -i $EXECUTIONID
echo "get deploy completed Successfully"
echo " "
 
echo "*************************************************"

Currently, this feature is in Beta and behind the feature flag GIT_SIMPLIFICATION. Contact Harness Support to enable the feature.​

Simplified Git Experience integrates your Harness Pipelines and resources with your Git repos. Simplified Git Experience enables you to do the following:

  • Save a Pipeline and its related Entities, such as Input Sets, in a Git repo rather than the Harness database. Git becomes the "single source of truth" for the Pipeline. Every time you update and save a Pipeline in the Pipeline Studio, Harness checks in the changes into your Git repo.
  • Maintain separate versions of your Pipelines and related configs in forks and non-default branches.
  • Maintain your codebase and Harness Pipelines in the same Git repo and branch. For example, you can create a release branch with a Harness Pipeline specifically configured for that release.
  • You can also maintain your codebase and Harness Pipelines in separate repos, with parallel branches to synchronize your updates.
  • Merge Pipeline changes between branches the same way you can with code changes.
  • Create Triggers that launch builds when your Pipelines get updated.
  • Customize how your Harness config YAMLs are organized in your repo. For example, you can organize your configs into harness-configs/pipelines, harness-configs/input-sets, and so on.

This topic describes how to set up Triggers and Input Sets in a Pipeline that has Simplified Git Experience enabled.

Before You Begin

You'll need the following:

  • A Git repo with at least one branch and a Personal Access Token (PAT) for your account. Harness needs the PAT to use the Git platform APIs. The PAT is encrypted using a Harness Secret Manager. Your Git Personal Access Token is stored in your Harness secret and is a private key to which only you have access. This secret cannot be accessed or referenced by any other user.
    Make sure your repo has at least one branch, such as main or master. For most Git providers, you simply add a README file to the repo, and the branch is created.
  • A Harness Pipeline with Simplified Git Experience enabled. In this how-to, you will cross-check your updates in both your codebase and in the Harness UI. See New Harness Git Experience QuickStart.

Step 1: Select the Branch

When you edit your Pipeline in the Harness UI, you are editing a branched version of that Pipeline. Make sure that you are editing in the correct branch. You can switch branches using the branch picker in the top left.

"The Pipeline does not exist in the selected branch:" What to do

If you try to switch into a branch that does not include the Pipeline definition, you will get a message like this:

If you see this message, add and commit a version of the Pipeline YAML to the target branch in your repo.

You can also add the Pipeline in the Harness UI:

  1. Select a branch that includes the Pipeline YAML.
  2. Edit your Pipeline with the changes you want to check in to the new branch.
  3. Click Save > Save Pipeline.
  4. In the Save Pipelines dialog, select the branch where you want to save your changes.
    You can save to an existing branch or a new branch. You can also choose to save your changes to one branch and then merge them into another.
  5. Click Save. Harness saves the updated Pipeline in the selected branch, or creates a Pull Request with your changes.
  6. If needed, go to your codebase and merge the Pull Request.

Step 2: Set the Codebase (Optional)

When you set up your Pipeline, you specify the Git repo where you want to save your Harness Pipeline configs. This is the default codebase for your Pipeline.

If you want to maintain your Harness configs and codebase in different repos, click the Codebase button (right) and set up the Git repo for your codes. See Create and Configure a Codebase.

Step 3: Create an Input Set and Save It to Your Branch

When Simplified Git Experience is enabled, your Input Set configs get saved in your repo and branch along with your Pipeline configs. Before you create a Trigger, you need to create one or more Input Sets to use with your Triggers.

In the Pipeline Editor, check the branch-selector pulldown and verify that you are in the correct branch.

Click Input Sets and then +New Input Set.

Specify the Name, Yaml Path, and CI Codebase fields as needed for the selected branch.

The YAML path should point to a file under the .harness folder in your repo. For example, .harness/input-sets/myInputSet.yaml.

See also Run Pipelines using Input Sets and Overlays and Built-in CI Codebase Variables Reference.

You are defining the Input Set for a specific branch in a specific repo. For this reason, the Select Git Connector, Repository, and Git Branch fields are displayed for informational purposes but are not editable.

Click Save and select the branch for the Input Set.

Step 4: Create a Trigger

Unlike Pipelines and Input Sets, Triggers are saved in the Harness database and apply to all branches in the codebase. A Trigger receives all webhook payloads sent to its URL. The Trigger processes each payload and starts a build if the payload matches the Conditions defined for that Trigger.
